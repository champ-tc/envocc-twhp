ระบบการประเมิน Cover — API Reference (ภาษาไทย)

การตรวจประเมินแบบลำดับชั้น โดยมี ODPC เป็นผู้ตัดสินสุดท้าย · อ้างอิง CONTEXT.md, ADR-0003, ADR-0004, ADR-3 (admin) · อัปเดต bolt-006 ถึง bolt-012

1ผู้ประเมินและการแบ่งหมวด

ผู้ประเมิน (Evaluator) แต่ละคนถูกจำกัดด้วย region (เห็นเฉพาะ Cover ในภูมิภาคของตน) และ level ซึ่งกำหนดหมวดที่ตรวจและอำนาจ การตรวจเป็นแบบลำดับชั้น ไม่ใช่แบบเท่าเทียม

Mental Tier-1

หมวด: Mental

ตัดสินได้เฉพาะหมวดของตน · ไม่จบกระบวนการ (non-finalizing) — Cover คง in_review

DOH Tier-1

หมวด: Disease, Safety

ตัดสินได้เฉพาะหมวดของตน · ไม่จบกระบวนการ — แก้ไขผลของตนเองได้ขณะ Cover ยัง in_review

ODPC Finalizer

หมวดหลัก: Collaborate, Outcome

อำนาจเหนือทุกหมวด — แก้ผลที่ยังไม่ finished ได้ · เป็นผู้เดียวที่ปิดงาน (เขียน coverLogs + คำนวณเกรด + ส่งอีเมล)

Admin DOED

ทำหน้าที่เป็น ODPC ระดับประเทศ

อำนาจเท่า ODPC ทุกประการ แต่ไม่จำกัด region — เข้าถึงและปิดงาน Cover ได้ทุกเขต ผ่าน /admin/covers/*

กฎ override: Tier-1 แก้ได้เฉพาะผลของตนขณะ in_review · ODPC แก้ได้ทุกรายการที่ยังไม่ finished · คำตอบที่ finished แล้วล็อกถาวร — แม้แต่ ODPC ก็แก้ไม่ได้
Admin-as-Evaluator: ผู้ดูแลระบบ (role DOED) เข้าร่วม flow เดียวกับ ODPC ได้ผ่าน /twhp/api/admin/covers/{coverId}/{answers,verdict} (guard: adminGuard) โดยระบบสร้าง context { level: "ODPC", region: null } ให้อัตโนมัติ — ใช้ ODPC commit path เดิมทั้งหมด (override · backstop · finalize · grade · email) · audit ด้วย accountId ของแอดมินในคอลัมน์ evaluation_id/evaluator_id เดิม (ไม่เปลี่ยน schema)
Two-finalizer (ADR-3): เมื่อมีทั้ง ODPC ประจำเขตและแอดมินระดับประเทศ Cover หนึ่งใบอาจมีผู้ปิดงานได้ 2 ทาง — v1 ไม่ล็อก เพราะคำตอบ finished ล็อกถาวรอยู่แล้ว และ finalize gate กันการ commit ซ้ำ (last-commit-wins)

2ผลการตัดสิน 3 แบบต่อหนึ่งคำตอบ

ผลการตัดสินสถานะคำตอบverdict_choicedescriptionไฟล์
Approve อนุมัติ Tier-1 → recommended
ODPC → finished		 — ไม่บังคับ คงไว้
Change-score ปรับคะแนน rejected 0–3 (ไม่มี n/a) บังคับ คงไว้ (โรงงานใช้คัดค้าน)

Answer status enum (4 ค่า): in_review · recommended · rejected · finished

4รอบเจรจาคะแนน (Negotiation Loop)

เมื่อ ODPC ตีกลับ Cover เป็น in_progress โรงงานต้องจัดการแต่ละคำตอบที่ rejected ด้วย POST /answers/negotiate:

กรณีactionเงื่อนไขผลลัพธ์
Change-score
(มี verdict_choice)		 "accept" ไม่ต้องส่ง selectedChoice ใหม่ · ตรวจไฟล์ตาม verdict_choice
ถ้า verdict_choice สูงกว่าเดิมต้องมีไฟล์ถึง level นั้น		 คำตอบ → recommended
selectedChoice อัปเดตเป็น verdict_choice
"redo" ส่ง selectedChoice + ไฟล์ใหม่ · ตรวจไฟล์ตาม selectedChoice ใหม่ คำตอบ → in_review
Reject
(verdict_choice = null)		 "redo" ส่ง selectedChoice + ไฟล์ใหม่ · ห้าม accept (ไม่มี verdict_choice) คำตอบ → in_review
Re-submit gate: โรงงานกด POST /submission ได้ก็ต่อเมื่อไม่มีคำตอบใดเป็น rejected — ต้อง accept หรือ redo ครบก่อน ถ้า Cover เป็น finished ส่งใหม่ไม่ได้เลย
Live Choice: คะแนนที่ใช้คำนวณสกอร์ = answers.selectedChoice เสมอ — อัปเดตอัตโนมัติเมื่อโรงงาน accept · ดังนั้น open verdict (ยังไม่ accept) ไม่กระทบคะแนนรายงาน

5เกรด/ระดับรางวัล (Grade)

คำนวณสดทุกครั้ง (ไม่บันทึกถาวร) จาก live choice ของทุกคำตอบ โดยใช้ลำดับจากบนลงล่าง — ใช้ระดับแรกที่ผ่านทุกเงื่อนไข:

ระดับเงื่อนไข (ต้องครบทุกข้อ)
🥇gold
 ทุกหมวด > 80% · คะแนนรวม ≥ 90% · ทุกข้อที่ special > 0 ได้คะแนน "3"
🥈silver
 ทุกหมวด > 60% · คะแนนรวม ≥ 80%
📜certificate
 คะแนนรวม ≥ 60%
🎗️joined
 คะแนนรวม < 60%
Silver→Certificate cliff: ถ้าหมวดใด ≤ 60% จะตกมา Certificate แม้คะแนนรวมจะสูงมาก เช่น รวม 90% แต่หมวดหนึ่งได้ 55% → Certificate

grade ปรากฏใน: response ของ POST /verdict (ODPC เท่านั้น) + ทุก score endpoint (null ถ้า Cover ยังไม่ finished)

8Schema อ้างอิง

ScoreGroup

{ scoredCount: number, maxScore: number, achievedScore: number, percentage: 0–100 }

percentage = Math.round(achievedScore / maxScore * 100) · n/a ไม่นับใน denominator

Grade

Union literal: "gold" | "silver" | "certificate" | "joined" — หรือ null ถ้า Cover ยังไม่ finished

Answer Status

Enum: "in_review" · "recommended" · "rejected" · "finished"

Cover Status

Derived จาก latest coverLogs: "in_progress" · "in_review" · "finished"
